# MEL-003

## Resumo do problema

O backend de tarefas ja estava disponivel, mas o frontend ainda nao expunha uma superficie minima para uso operacional de `task`.

## Causa raiz

A frente foi entregue em etapas de baixo risco: primeiro schema, depois CRUD HTTP e, nesta rodada, a UI minima.

## Arquivos alterados

- `ROADMAP.md`
- `apps/web/src/App.tsx`
- `apps/web/src/components/TaskUserPanel.tsx`
- `apps/web/src/components/admin/AdminTasksPanel.tsx`
- `apps/web/src/components/tasks/taskUi.ts`
- `apps/web/src/components/AdminDashboard.tsx`
- `apps/web/src/components/admin/AdminSidebar.tsx`
- `apps/web/src/components/admin/types.ts`
- `apps/web/src/types.ts`
- `apps/web/src/App.test.tsx`
- `apps/web/src/components/TaskUserPanel.test.tsx`
- `apps/web/src/components/admin/AdminTasksPanel.test.tsx`

## Estrategia da correcao

- adicionar rota dedicada `/tasks` no fluxo do usuario
- expor painel minimo de tarefas para user e admin
- reutilizar o cliente HTTP ja validado em `MEL-002`
- compartilhar labels, badges e formatacao de datas em helper proprio
- validar a nova superficie com testes focados de componente e roteamento

## Riscos considerados

- nao alterar notificacoes, lembretes ou scheduler nesta rodada
- nao introduzir board, comentarios ou realtime de tarefas
- manter a UI restrita a lista/detalhe simples para reduzir superficie de regressao
