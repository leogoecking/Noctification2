# REF-009

## Resumo do problema

A suíte web tinha arquivos de teste grandes demais, com fixtures base e setups padrão repetidos em `AdminTasksPanel`, `AdminDashboard` e `AprPage`. Isso elevava o custo de manutenção do harness de testes e dificultava mudanças locais de baixo risco.

## Causa raiz

- Fixtures e respostas padrão foram sendo declaradas dentro dos próprios arquivos `.test.tsx`.
- O dashboard administrativo concentrava um `beforeEach` extenso com payloads duplicados.
- O módulo APR mantinha datasets-base grandes no próprio arquivo de teste.

## Arquivos alterados

- `apps/web/src/test/fixtures.ts`
- `apps/web/src/test/adminTasksPanelFixtures.ts`
- `apps/web/src/test/adminDashboardFixtures.ts`
- `apps/web/src/test/aprFixtures.ts`
- `apps/web/src/features/tasks/test/AdminTasksPanel.test.tsx`
- `apps/web/src/components/AdminDashboard.test.tsx`
- `apps/web/src/features/apr/AprPage.test.tsx`

## Estratégia da correção

- promover builders reutilizáveis para `fixtures.ts`;
- extrair helpers específicos por suíte para respostas padrão e datasets-base;
- reduzir duplicação sem alterar cenários, asserções nem contratos do `api`;
- manter as mudanças restritas ao harness de testes.

## Riscos considerados

- tipagem incorreta dos helpers frente às respostas reais do `api`;
- perda de legibilidade se a extração escondesse demais o cenário de teste.

## Validação executada

- `npm run test --workspace=apps/web -- AprPage`
- `npm run test --workspace=apps/web -- AdminDashboard`
- `npm run test --workspace=apps/web -- AdminTasksPanel`
- `npm run test --workspace=apps/web`
- `npm run typecheck --workspace @noctification/web`
